# Кратко описание логики БД
## Регистрация пользователя
### user
Таблица users хранит пользователей платформы. 
Основные поля: id, role, phone, created_at, blocked_at. 
Связи: пользователь может создать много кампаний, участвовать в сделках, иметь подписки и отзывы. 
Важное ограничение: phone должен быть уникальным, для того, чтобы у одного пользователя можно было добавить 2 роли, поля is_blogger, is_seller не выносим их в отдельный справочник

### privacy_policy
Для хранения офферты и согласия об обработке данных, которые ддействовали на момент регистрации. Дата подписания = дата регистрации в таблице user

## Селлер создает кампанию
### campaigns
Настройки храним как гибкие - поле setting_json

## Блогер заполняет профиль
### user_platforms
Platform и region жестко заданные enum, audience и reach реализуем через справочники, для возможности дальнейшей донастройки

### topics
У одной платформы может быть несколько тематик, для возможности дальнейшего расширения - реализуем через справочних. Связь многие ко многим организуем через промежуточную таблицу topic_user_platform. Таже таблицу topics используем при создании рекламмной акции для выбора тематики. Связь topics с campaigns один к одному.

## Система отправляет оффер блогеру
### offers
Принято решение хранить офферы отдельно от сделок, чтобы не перегружать таблицу сделок, которая крайне важна и историю хранить нужно достаточно длительно, а отклоненные офферы можно чистить с периодичностью.

### deals
Один оффер - одна сделка, связь один к одному по id_оффера, поле is_safe_deal для безопасных сделок, поля campaign_id и blogger_id дублируем для удобства отображения актуальных сделок, чтобы не использовать сложные запросы

## Сделка меняет статусы
### status_log
Историю статусов таблиц users, campains, offers, deals, safe_deal_payments, user_platforms храним в status log, фиксируем кто изменил статус, когда и актуален ли статус на данное время (is_active) для разбора возможных споров. Из какой таблицы статус - entity_name, какой записи принадлежит - entity_id/

## Safe-сделка и платеж
### safe_deal_payments
Безопасная сделка - поле is_safe_deal в таблице deals. Связь с таблице deals один ко многим, потому что предусмотрена повторная оплата, в случае неудачи. Храним количество попыток оплаты, код ошибки от системы и текст ошибки. Внешний payment_id для идентификации платежа, должен быть уникальным, чтобы не списать оплату дважды. Payment_method - все системы предусмотренные для оплаты (enum).

## Подписка блогера
### subscriptions
Подписки пользователя. Связь пользователя с подписками один ко многим, потому что может быть несколько подписок. Храним дату оформления подписки, когда стартует, когда заканчивается, по какому тарифу была оформлена, какая применена скидка, сколько финальная стоимость.

### subscription_templates
Шаблон тарифа. Храним дату создания, актуальность, базовую стоимость, на какой период действует. Один тариф может быть во многих подписках, связь один ко многим

## Промокоды
### promos
Условия промокода хранима гибкой структурой - rules_json, списки тарифов на которые распространяется - массив идентификаторов template_ids. Один промокод может быть применен в нескольких подписках

## Файлы
### files
Файлы принадлежат конкретному пользователю (один ко многим). Один и тот же файл моет быть использован в нескольких кампаниях (многие ко многим), связь организуем через таблицу files_campaign. ссылку на файл храним в storage_path.

## Отзывы
### reviews
Связь со сделкой - один ко многим, отзыв может оставить как селлер, так и блогер. Нужно ограничить, что каждая сторона может оставить не более одного отзыва (составной ключ deals_id,user_id). Храним оценку rating, пересчитываем rating_avg и rating_count в таблице users один раз в сутки процедурой или реализуем триггер в момент вставки (?).

## Сообщения и уведомления
Сообщения и уведомления разделены, чаты необходимо хранить длительно, после завершения сделки, они жестко связаны с таблицей deals.
### notifications
В таблице поля entity_name и entity_id чтобы из уведомления по сделке или офферу можно было сразу перейти на нужную страницу

### messages
Таблицу привязываем к deals через промежуточную таблицу chats, в случае разбора спора и необходимости подключения третьей стороны, потребуется создать еще чаты с продавцом и блогером, которые должны быть привязаны к сделке. В message храним автора сообщения, дату создания и идентификатор прочтения.
