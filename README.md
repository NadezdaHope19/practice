## Документ с требованиями мобильному приложению Adspring

### 1. **Бизнес-требования (Business Requirements)**  

BR-01 Увеличить количество активных блогеров на 20% за 4 месяца за счёт удобного мобильного приложения. Успех измерим по количеству оплаченных подписок блоггеров из приложения.

BR-02 Сократить время создания рекламной кампании на 30% за счет удобного процесса для селлера (даже в отсутствие интернета), увеличить поток пользователей сервиса и сократить временные затраты на поиск исполнителя. Успех оценим по количеству новых регистраций селлеров.

BR-03 Снизить время реагирования блоггера на оффер за счет отправки моментального push-уведомления.

BR-04 Повысить конверсию из оффера в заключённую сделку за счет удобного интерфейса и push-уведомлений. Успех измерим по росту числа завершённых сделок.

BR-05 Привлечь новую аудиторию блогеров, которые не пользуются веб-версией и tg-версией, которые ценят удобство мобильной версии.

BR-06 Обеспечить безопасный и удобный процесс оплаты и реализовать функционал безопасной сделки с привязкой карты, что повысит доверие к сервису.

BR-07 Обеспечить формирование еженедельной отчетности для бизнеса о количестве зарегистрированных пользователей, оплаченных подписок и количестве завершенных сделок и направлять на электронную почту руководителя отдела.

Входит в MVP:
#### Регистрация и онбординг
- Регистрация / вход по номеру телефона или через Telegram
- Выбор роли (селлер/ блогер)
- Онбординг-экраны (3-5 слайдов о возможностях)
- Принятие оферты и согласия на обработку персональных данных

#### Функционал селлера
- Создание рекламной кампании: название, описание, категория, бюджет (1000 – 500000 руб.), тип оплаты (бюджет / бартер), загрузка фото/видео товара
- Запуск кампании (поиск исполнителя)
- Просмотр входящих офферов от подобранных блогеров, push-уведомления
- Принятие/ отклонение оффера
- Подтверждение завершения работы блогера и оплата/отправка бартера
- Оценка блогера (по 5-балльной шкале) + текстовый комментарий

#### Функционал блогера
- Добавление социальных сетей (Instagram, Telegram, YouTube, VK, TikTok, Dzen): выбор платформы, ссылка, количество подписчиков, охваты, регион, тема
- Запрос на верификацию соцсети (генерация кода для вставки в профиль)
- Оформление подписки (месяц / 3 месяца / год) — через in-app purchase или эквайринг
- Переключатель «Готов получать заказы»
- Получение push-уведомлений с офферами, сообщением о продлении подписки
- Принятие/ отклонение оффера
- Подтверждение выполнение сделки кнопкой
- Привязка банковской карты для получения выплат
- Просмотр истории сделок и заработка
- Оценка селлера (по 5-балльной шкале) + текстовый комментарий

#### Общий функционал
- Личный кабинет (аватар, имя, рейтинг, статистика, отзывы)
- Чат внутри сделки (текст, отправка изображений)
- Push-уведомления на события: новый оффер, изменение статуса сделки, новое сообщение в чате, окончание подписки
- Предпросмотр документов офферты и согласия на обработку персональных данных
- Техническая поддержка (ссылка на FAQ / чат поддержки)

Остаётся на следующие версии:

- Возможность выбора английского языка
- Реферальная программа (скидка на оплату подписки за привлечение пользователей)
- Шаблоны рекламных кампаний, настраиваемые селлером
- Встроенный видеоредактор для обрезки видео кампаний непосредственно в приложении
- Тёмная тема
- Топ селлеров / топ блогеров
- Возможность менять тип пользователя внутри одного аккаунта


### 2. **Функциональные требования (Functional Requirements)**


#### Регистрация/аутентификация пользователя, выбор роли

FR-01 Регистрация пользователя осуществляется с помощью телефона, на который высылается код для подтверждения регистрации. После успешного ввода кода пользователь переходит к следующему этапу.

FR-02 При регистрации пользователь обязательно должен выбрать роль (блогер/селлер). Профиль пользователя формируется в соответствии с выбранной ролью.

FR-03 Для завершения регистрации пользователь должен подписать оферту и дать согласие на обработку персональных данных, для чего он проставляет галки в чекбоксах "Согласен с офертой" и "Согласен на обработку персональных данных". Только после этого кнопка "Завершить регистрацию" становится активной.

FR-04 Пользователь должен иметь возможность войти в свой аккаунт, созданный в tg-версии сервиса используя логин и пароль.


#### Личный кабинет (блогер)

FR-05 Блогер должен заполнить профиль: выбрать платформу соцсети (список - Instagram, Telegram, YouTube, VKontakte, TikTok, Dzen), ссылку на соцсеть, кол-во подписчиков, средние охваты, регион, тема блога (путешествия, обзоры техники, бьюти, мода, здоровье и фитнес, food-блог, финансы, творчество, юмор, it и тд). Все поля обязательны для заполнения.

FR-06 После заполнения информации, блогер запускает верификацию соцсети. Система должна сгенерировать уникальный 4-х (6-ти) значный код, который блогер должен разместить в описании профиля соцсети для проверки модератором. Статус соцсети устанавливается "На проверке".

FR-07 Запрос на проверку соцсети поступает модератору. Модератор: подтверждает соцсеть - блогеру направляется push-уведомление "Соцсеть верифицирована", в профиле напротив соцсети отобразить зеленый маркер (галку) или отклоняет - блогеру поступает push-уведомление с причиной отказа в верификации "Условия верификации не выполнены"

FR-08 После верификации хотя бы одной соцсети блогеру становится доступна кнопка "Оформить подписку" для возможности получения блогером офферов. Без активной подписки и верифицированной соцсети пользователь не участвует в поиске селлером исполнителя.

FR-09 Блогер привязывает банковскую карту, для осуществления безопасной сделки.


#### Добавить/удалить/редактировать соц. сеть (блогер)

FR-09 Блогер может добавить новую соцсеть, нажатием кнопки "Добавить соцсеть". Список соцсетей отображать в профиле блогера в хронологическом порядке (последняя добавленная сеть отображается в конце списка)

FR-10 Блогер может редактировать добавленную соцсеть, изменяя поля Количество подписчиков, Охваты, Регион, Тема блога

FR-11 Блогер может удалить соцсеть из профиля


#### Оформление подписки (блогер)

FR-12 Кнопка "Оформить подписку" вызывает форму с информацией о подписках (на месяц, 3 месяца, год) и стоимостью. Оплата выбранного тарифа осуществляется через эквайринг. После успешной оплаты в профиле блогера статус подписки отображается "Активна", желательно указывать дату окончания подписки.

FR-13 После оплаты подписки и при наличии хотя бы одной верифицированной соцсети блогеру становится доступен переключатель "Готов получать заказы", который, при включении, активирует блогера для участия в поиске селлером исполнителя.


#### Создание/управление рекламной кампанией (селлер)

FR-14 Селлер создает рекламную кампанию, заполняя следующие параметры: название, описание, категория, бюджет (от 1000 до 500000 руб), тип оплаты (бартер/бюджет), фото/видео товара. Заполненная карточка рекламной кампании сохраняется со статусом "Кампания создана".

FR-15 Селлер может редактировать кампанию в статусе "Кампания создана", уточняя необходимые параметры.

FR-16 Для поиска исполнителя, селлер должен запустить рекламную кампанию. Статус карточки меняется на "Поиск исполнителя".

FR-17 После назначения исполнителей кампании, она переходит в статус "В работе".

FR-18 После оплаты всех сделок по данной рекламмной кампании, она переходит в статус "Завершена".

FR-19 Селлер может повторить завершенную рекламную кампанию, не повторяя заполнение карточки.


#### Офферы и сделки (общие)

FR-20 Система, подобрав подходящих для рекламной кампании блогеров, направляет push-уведомление блогеру. При нажатии на уведомление, блогер может просмотреть информацию о кампании и принять или отклонить предложение.

FR-21 Блогер может установить фильтр в личном кабинете на получение офферов (не получать офферы по бартеру, не получать офферы с темой "Детские товары", "Товары для взрослых"), чтобы исключить свой аккаунт из поиска.

FR-22 Селлер может просмотреть список откликов на рекламную кампанию (платформа, количество подписчиков, охваты, рейтинг блогера).

FR-23 Селлер может принять отклик блогера и назначить исполнителя, статус сделки изменится на "Сделка создана".

FR-24 Блогер и Селлер получают доступ к чату по сделке.

FR-25 Блогер получает push-уведомление об изменении статуса сделки.

FR-26 Селлер и блогер должны иметь возможность отменить сделку, в случае неисполнения сроков второй стороны.


#### Безопасная сделка (общие)

FR-26 В чате блогер и селлер уточняют детали рекламмной акции. Переписка должна быть сохранена для решения спорных ситуаций.

FR-27 Селлер должен оплатить сделку для того, чтобы блоггер приступил к исполнению. Оплата замораживается платформой.

FR-28 Блогер получает push-уведомление об оплате селлером сделки и приступает к исполнению. Статус сделки меняется на "Оплачено".

FR-29 Блогер подтверждает исполнение сделки и прилагает скрины или ссылку на рекламу. Статус сделки меняется на "На проверке".

FR-30 Селлер получает push-уведомление об исполнении сделки блогером, проверяет исполнение и подтверждает исполнение. Статус сделки меняется на "Исполнено".

FR-31 Система автоматически переводит оплату сделки блогеру на привязанную карту.


#### Оценка и отзывы (общие)

FR-32 Селлер должен оценить работу блогера по 5-балльной шкале по предложенным критериям: качество контента, скорость, коммуникация и написать текстовый отзыв.

FR-33 Блогер должен оценить селлера по 5-балльной шкале по предложенным критериям: качество продукта, своевременная оплата, скорость коммуникации и написать текстовый отзыв.

FR-34 Селлер и блогер могут подать жалобу, на действия второй стороны, для оперативного реагирования сервиса на недобросовестные действия в рамках сделки.

FR-35 Средняя оценка и отзывы должны отображаться в профиле и обновляются не реже, чем раз в сутки


#### Уведомления (общие)

FR-35 Приложение должно обеспечивать отправку push-уведомлений блогеру о получении офферов, любых изменениях статусов сделки, о новом сообщении по сделке, об оканчивающейся подписке, об истечении срока 7 дней на выполнение сделки.

FR-36 Приложение должно обеспечивать отправку push-уведомлений селлеру о поступающих откликах, любых изменениях статусов сделки, о новом сообщении по сделке, о превышении сроков сделки блогером более чем на 7 дней.

FR-37 При нажатии на push-уведомление с оффером - открывается окно с деталями оффера, об изменении статуса сделки - переводит в список сделок, о новом сообщении - в чат по сделке.


### 3. **Нефункциональные требования (Non-Functional Requirements)**  

#### Производительность

NFR-01 Холодный старт не должен превышать 3 секунд.

NFR-02 Время отклика системы на запросы пользователя не должно превышать 2 секунд, даже при максимальной нагрузке.

NFR-03 Все push-уведомления должны доставляться не позже чем через 10 секунд после события.


#### Надежность

NFR-04 Обеспечить долю сессий без падений > 99,5%

NFR-05 Система должна быть доступна  >= 99,9%

NFR-06 Обеспечить обработку ошибок сети и понятное сообщение для пользователя в случае возникновения ошибки. 

NFR-07 Время устранения критической ошибки обеспечить в пределах 2 часов.

NFR-08 Обеспечить восстановление из бэкапов данных профилей и сделок.

NFR-09 Обеспечить корректную работу при прерываниях работы, например во время звонка, потери сети. Сохранять данные и синхронизировать их позже.


#### Безопасность

NFR-10 Все данные между клиентом и сервером должны передаваться по безопасному протоколу https для обеспечения безопасности.

NFR-11 Все локальные данные (пароли, токены) должны храниться в зашифрованном виде

NFR-12 Платежная информация не хранится, только токены.

NFR-13 Приложение должно поддерживать вход по отпечатку пальца (Touch ID / Face ID в iOS, BiometricPrompt в Android)

### 4. **Требования к пользовательскому интерфейсу (UI Requirements)**  
   
   - Структура экранов (можно в виде списка или простой схемы).  
   - Ключевые элементы (кнопки, формы, модальные окна).

UI-01 Приложение должно следовать Human Interface Guidelines для iOS и Material Design 3 для Android.

UI-02 Цветовую палитру использовать, разработанную для web-версии приложения.

UI-03 Контрастность текста и фона не менее 4,5:1

UI-02 Элементы интерфейса не должны обрезаться на разных размерах экранов или при разной системной ориентации (вертикальной/горизонтальной).

UI-03 Текст должен быть четким, а размер шрифта должен соответствовать системным настройкам доступности для слабовидящих.

UI-04 Интерфейс приложения должен быть адаптивным и корректно отображаться на устройствах с разным разрешением экрана.

UI-05 Важные уведомления системы должны отображаться на всех страницах в виде всплывающего окна или уведомления.

UI-06 Структура экранов:

- регистрация и онбординг
- главный экран с табами: профиль (блогер), Список рекламных кампаний(селлер)/Список соцсетей(блогер), Мои сделки, Чаты, Список офферов(блогер)/Список откликов(селлер)
- карточка соцсети (блогер) для добавления/редактирования
- карточка рекламной кампании (селлер) для создания/редактирования
- оформить подписку (блогер)
- карточка оффера
- оценка и отзыв

UI-07 Регистрация и онбординг: 
- при первом запуске приложения приветствие и 2-3 слайда с иллюстрациями, кратко объясняющие ценность для пользователя
- форма ввода логина(телефона) и пароля или регистрации (чекбокс, если нужна регистрация), позволяет или войти в имеющийся аккаунт или создать новый. Ввод телефона осуществлять по маске. Если установлена галка "регистрация", то сделать активным выбор роли пользователя - блогер иди селлер. Кнопка "зарегистрироваться" активна только после выбора роли и введения кода, высланного на телефон.
- если пользователь уже зарегистрирован, то после ввода логина/пароля приложение должно автоматически определить роль.

UI-08 Главный экран - Профиль(блогер)содержит: 
- аватар блогера, среднюю оценку, отзывы, баннер "Верифицирован", если пройдена верификация соцсети.
- содержит информацию об активной подписке и дате до которой действует подписка (зеленый баннер "Подписка активна до хх.хх.хххх") или кнопку "оформить подписку", если подписки нет. Кнопка доступна только при неактивной подписке; 
- кнопку "добавить соцсеть";
- переключатель "Готов получать заказы"; 
- кнопку "привязать карту" для обеспечения безопасной сделки;
- фильтры (чекбоксы) для настройки получения офферов: только с оплатой сделки (исключая бартер), исключить категории товаров (...)

UI-09 При нажатии на кнопку "добавить соцсеть" открывается карточка, в которой нужно выбрать платформу соцсети (выпадающий список - Instagram, Telegram, YouTube, VKontakte, TikTok, Dzen), ссылку на соцсеть, кол-во подписчиков, средние охваты, регион, тема блога (выпадающий список: путешествия, обзоры техники, бьюти, мода, здоровье и фитнес, food-блог, финансы, творчество, юмор, it и тд. Все поля обязательны для заполнения. Внизу формы кнопки "добавить", "верифицировать".

UI-10 При нажатии на кнопку "оформить подписку" открывается карточка, в которой можно осуществить выбор длительности подписки - месяц, 3 месяца, год и нажать кнопку "оплатить", варианты оплаты - через T-Bank и CloudPayments

UI-11 При нажатии кнопки "привязать карту" открывается форма для добавления карты.

UI-12 Главный экран - Список соцсетей (блогер). список соцсетей отсортирован по хронологии добавления: название платформы, количество подписчиков, статус верификации. кнопки для редактирования соцсети или удаления.

UI-13 Главный экран - Профиль(селлер) содержит:
- среднюю оценку, отзывы
- кнопку "создать рекламную кампанию"

UI-14 При нажатии на кнопку "создать рекламную кампанию" открывается карточка рекламной кампании с полями "название, описание, категория, бюджет, тип оплаты (список: бартер/бюджет), фото/видео товара", кнопку "создать рекламную кампанию"

UI-15 Главный экран - Список рекламных кампаний (селлер). Список отсортирован по хронологии добавления: название, категория, статус (кампания создана, поиск исполнителя, в работе, завершена)

UI-16 Главный экран - Список офферов (блогер): содержит все неоработанные офферы, отсортированные по хронологии от самого свежего: наименование селлера, товар, бюджет, статус (не просмотрен, принят).после подтверждения селлером сделки, оффер перемещается во вкладку "Мои сделки"

UI-17 При нажатии на оффер открывается карточка с бюджетом, описанием товара и кнопками “Принять” / “Отклонить”. 

UI-18 Главный экран - Список откликов (селлер): содержит все неотработанные отклики отсортированные по хронологии от самого свежего:  пдатформа соцсети, количество подписчиков, охваты, статус (не просмотрен, принят). после принятия селлером отклика, отклик перемещается во вкладку "Мои сделки"

UI-19 При нажатии на отклик открывается карточка отклика с указанием имени блогера, платформа соцсети, количество подписчиков, охваты, оценка.

UI-20 Главный экран - Мои сделки (общий): содержит список всех сделок с указанием имени блогера/селлера, товара, даты заключения сделки, их статуса (), 


### 5. **Требования к данным (Data Requirements)**  
   - Какие данные кэшируются локально.  
   - Стратегия синхронизации с сервером.  
   - Объёмы данных, пагинация.  
DR-01 вся информация о заказах должна быть доступна для отчетности через субд храниться в базе данных в течение не менее 5 лет для возможности анализа и составления отчетности.

DR-02 для всех изменений данных должен вестись журнал, фиксирующий дату, время, пользователя совершившего изменение

DR-03 данные о пользователях должны храниться в защищенной базе данных с регулярным резервным копированием

DR-04 Локально кэшируются список активных сделок и названия кампаний (обновление при каждом открытии). Фото товаров не кэшируются — подгружаются из сети. При отсутствии интернета показывается последняя актуальная версия с пометкой “Офлайн”

DR-05 структура бд должна быть оптимизирована для быстрого выполнения запросов и эффективного хранения информации.


### 6. **Ограничения и допущения (Constraints & Assumptions)**  

CA-01 Приложение должно работать через существующий бэкенд, который мы не меняем.

CA-02 Предполагаем, что все пользователи имеют стабильный интернет большую часть времени.
   
CA-02 Блогер не может добавить более 10 социальных сетей.

CA-03 Селлер не может создать кампанию с бюджетом менее 1000 руб и более 500000 руб.

CA-04 Селлер не может менять карточку активной рекламной акции.

CA-0 Блогер не может участвовать в поиске с неактивной подпиской или не имея хотя бы одной верифицированной соцсети.

CA-0 Блогер исключается из поиска, если у него в работе 50 незаконченных сделок.

CA-0 Блогер не может удалить соцсеть, если она является единственной верифицированной.

CA-0 Селлер не пожет подтвердить выполнение сделки, если блогер не подтвердит исполнение и не приложит доказательства.

CA-0 Селлер и блогер не могут создать чат вне сделки.

CA-0 Селлер и блогер могу поставить оценку друг другу, только в рамках сделки. Если взаимодействия в рамках сделки не было - оценить невозможно.

CA-0 MVP должен быть выпущен в течение 3 месяцев.
