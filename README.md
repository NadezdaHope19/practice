## Документ с требованиями мобильному приложению Adspring

1. **Бизнес-требования (Business Requirements)**  

BR-01 Увеличить количество активных блогеров на 20% за 4 месяца за счёт удобного мобильного приложения. Успех измерим по количеству оплаченных подписок блоггеров из приложения.

BR-02 Сократить время создания рекламной кампании на 30% за счет удобного процесса для селлера (даже в отсутствие интернета), увеличить поток пользователей сервиса и сократить временные затраты на поиск исполнителя. Успех оценим по количеству новых регистраций селлеров.

BR-03 Снизить время реагирования блоггера на оффер за счет отправки моментального push-уведомления.

BR-04 Повысить конверсию из оффера в заключённую сделку за счет удобного интерфейса и push-уведомлений. Успех измерим по росту числа завершённых сделок.

BR-05 Привлечь новую аудиторию блогеров, которые не пользуются веб-версией и tg-версией, которые ценят удобство мобильной версии.

BR-06 Обеспечить безопасный и удобный процесс оплаты и реализовать функционал безопасной сделки с привязкой карты, что повысит доверие к сервису.

BR-07 Обеспечить формирование еженедельной отчетности для бизнеса о количестве зарегистрированных пользователей, оплаченных подписок и количестве завершенных сделок и направлять на электронную почту руководителя отдела.

2. **Функциональные требования (Functional Requirements)**

#### Регистрация/аутентификация пользователя, выбор роли

FR-01 Регистрация пользователя осуществляется с помощью телефона. Номер телефона вводится по маске, на который высылается код для подтверждения регистрации. После успешного ввода кода пользователь переходит к следующему этапу.

FR-02 При регистрации пользователь обязательно должен выбрать роль (блогер/селлер). Профиль пользователя формируется в соответствии с выбранной ролью.

FR-03 Для завершения регистрации пользователь должен подписать оферту и дать согласие на обработку персональных данных, для чего он проставляет галки в чекбоксах "Согласен с офертой" и "Согласен на обработку персональных данных". Только после этого кнопка "Завершить регистрацию" становится активной.

FR-04 Пользователь должен иметь возможность войти в свой аккаунт, созданный в tg-версии сервиса используя логин и пароль.

#### Личный кабинет (блогер)

FR-05 Блогер должен заполнить профиль: выбрать платформу соцсети (список - Instagram, Telegram, YouTube, VKontakte, TikTok, Dzen), ссылку на соцсеть, кол-во подписчиков, средние охваты, регион, тема блога (путешествия, обзоры техники, бьюти, мода, здоровье и фитнес, food-блог, финансы, творчество, юмор, it и тд). Все поля обязательны для заполнения.

FR-06 После заполнения информации, блогер запускает верификацию соцсети. Система должна сгенерировать уникальный 4-х (6-ти) значный код, который блогер должен разместить в описании профиля соцсети для проверки модератором. Статус соцсети устанавливается "На проверке".

FR-07 Запрос на проверку соцсети поступает модератору. Модератор: подтверждает соцсеть - блогеру направляется push-уведомление "Соцсеть верифицирована", в профиле напротив соцсети отобразить зеленый маркер (галку) или отклоняет - блогеру поступает push-уведомление с причиной отказа в верификации "Условия верификации не выполнены"

FR-08 После верификации хотя бы одной соцсети блогеру становится доступна кнопка "Оформить подписку" для возможности получения блогером офферов. Без активной подписки и верифицированной соцсети пользователь не участвует в поиске селлером исполнителя.

#### Добавить/удалить/редактировать соц. сеть (блогер)

FR-09 Блогер может добавить новую соцсеть, нажатием кнопки "Добавить соцсеть". Список соцсетей отображать в профиле блогера в хронологическом порядке (последняя добавленная сеть отображается в конце списка)

FR-10 Блогер может редактировать добавленную соцсеть, изменяя поля Количество подписчиков, Охваты, Регион, Тема блога

FR-11 Блогер может удалить соцсеть из профиля

#### Оформление подписки (блогер)

FR-12 Кнопка "Оформить подписку" вызывает форму с информацией о подписках (на месяц, 3 месяца, год) и стоимостью. Оплата выбранного тарифа осуществляется через эквайринг. После успешной оплаты в профиле блогера статус подписки отображается "Активна", желательно указывать дату окончания подписки.

FR-13 После оплаты подписки и при наличии хотя бы одной верифицированной соцсети блогеру становится доступен переключатель "Готов получать заказы", который, при включении, активирует блогера для участия в поиске селлером исполнителя.

#### Создание/управление рекламной кампанией (селлер)

FR-14 Селлер создает рекламную кампанию, заполняя следующие параметры: название, описание, категория, бюджет (от 1000 до 500000 руб), тип оплаты (бартер/бюджет), фото/видео товара. Заполненная карточка рекламной кампании сохраняется со статусом "Кампания создана".

FR-15 Селлер может редактировать кампанию в статусе "Кампания создана", уточняя необходимые параметры.

FR-16 Для поиска исполнителя, селлер должен запустить рекламную кампанию. Статус карточки меняется на "Поиск исполнителя".

3. **Нефункциональные требования (Non-Functional Requirements)**  
   - Производительность (например, экран должен загружаться менее чем за 2 секунды).  
   - Надёжность (доступность 99.9%, обработка сбоев).  
   - Безопасность (хранение токенов, биометрия, шифрование).  
   - Совместимость (версии iOS/Android).  
   - Удобство использования (минимальное количество кликов для ключевого сценария).  
   - Масштабируемость.  
NFR-01 Система должна быть защищена от несанкционированного доступа любыми методами, включая двухфакторную аутентификацию
Все данные передаваемые между клиентом и сервером должны быть зашифрованы протоколом данных https для обеспечения безопасности
NFR-02 время отклика системы на запросы пользователя не должно превышать 3 секунды даже при максимальной нагрузке

NFR-03 система должна быть совместима с разными браузерами

NFR-04 приложение должно быть легко масштабируемым для возможности добавления нового функционала и увеличения количества пользователей.

NFR-05 Время открытия экрана со списком кампаний не должно превышать 2 секунд даже при 500 активных кампаниях. Все push-уведомления должны доставляться не позже чем через 10 секунд после события.

4. **Требования к пользовательскому интерфейсу (UI Requirements)**  
   - Общие принципы (например, соответствие Material Design и Human Interface Guidelines).  
   - Структура экранов (можно в виде списка или простой схемы).  
   - Ключевые элементы (кнопки, формы, модальные окна).  
   - Требования к адаптивности и доступности (размер шрифта, контрастность).  
UI-01 Цветовая схема пользовательского интерфейса должна соответствовать 

UI-02 важные уведомления системы должны отображаться на всех страницах в виде всплывающего окна или уведомления

UI-03 все элементы управления должны иметь понятное описание и всплывающие подсказки для удобства пользователей

UI-04 размер и шрифт должны быть настроены таким образом чтобы обеспечивать хорошую читабельность на разных устройствах

UI-05 интерфейс приложения должен быть адаптивным и корректно отображаться на устройствах с разным разрешением экрана

UI-06 На главном экране блогера внизу — табы: “Предложения”, “Мои сделки”, “Соцсети”, “Профиль”. При нажатии на оффер открывается карточка с бюджетом, описанием товара и кнопками “Принять” / “Отклонить”.

5. **Требования к данным (Data Requirements)**  
   - Какие данные кэшируются локально.  
   - Стратегия синхронизации с сервером.  
   - Объёмы данных, пагинация.  
DR-01 вся информация о заказах должна быть доступна для отчетности через субд храниться в базе данных в течение не менее 5 лет для возможности анализа и составления отчетности.

DR-02 для всех изменений данных должен вестись журнал, фиксирующий дату, время, пользователя совершившего изменение

DR-03 данные о пользователях должны храниться в защищенной базе данных с регулярным резервным копированием

DR-04 Локально кэшируются список активных сделок и названия кампаний (обновление при каждом открытии). Фото товаров не кэшируются — подгружаются из сети. При отсутствии интернета показывается последняя актуальная версия с пометкой “Офлайн”

DR-05 структура бд должна быть оптимизирована для быстрого выполнения запросов и эффективного хранения информации.
