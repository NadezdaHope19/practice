## Документ с требованиями мобильному приложению Adspring

1. **Бизнес-требования (Business Requirements)**  
   - Цели внедрения мобильного приложения.  
   - Целевые метрики успеха (например, снижение времени создания кампании на 30%, рост числа активных блогеров на 20%).  
   - Границы проекта: что входит в MVP, что остаётся на следующие версии.  
BR-01 система должна автоматически формировать еженедельные отчеты о количестве зарегистрированных пользователей и количестве завершенных сделок и направляться на электронную почту руководителя отдела

BR-02 система должна предоставлять возможность настройки доступа для разных уровней пользователей

BR-03 разработка приложения должна обеспечить более удобный процесс оформления рекламной кампаний (даже в отсутствие интернета), увеличить поток пользователей сервиса и сократить временные затраты на поиск исполнителя

BR-01
Увеличить количество активных блогеров на 30% за 6 месяцев за счёт удобного мобильного приложения. Успех измерим по росту числа завершённых сделок и снижению отказов при создании кампании

BR-01

2. **Функциональные требования (Functional Requirements)**  
   - Разделить по ролям (селлер, блогер) и общим сценариям.  
   - Каждое требование должно быть **однозначным, конкретным, проверяемым**.  
   - Рекомендуется нумерация: FR-1, FR-2 и т.д.  
   - Обязательно описать: регистрация/вход, выбор роли, создание/управление кампаниями (для селлера), добавление соцсетей и верификация (для блогера), подписка, получение/обработка офферов, сделки (безопасные и обычные), отзывы, личный кабинет, уведомления.  
FR-01 пользователь должен иметь возможность войти в свой аккаунт используя логин и пароль
FR-02 приложение должно обеспечивать отправку уведомлений о предстоящих событиях

FR-03 Блогер может добавить новую соцсеть: выбрать платформу (YouTube/Instagram/Telegram), ввести ссылку на профиль, указать количество подписчиков и основные темы

3. **Нефункциональные требования (Non-Functional Requirements)**  
   - Производительность (например, экран должен загружаться менее чем за 2 секунды).  
   - Надёжность (доступность 99.9%, обработка сбоев).  
   - Безопасность (хранение токенов, биометрия, шифрование).  
   - Совместимость (версии iOS/Android).  
   - Удобство использования (минимальное количество кликов для ключевого сценария).  
   - Масштабируемость.  
NFR-01 Система должна быть защищена от несанкционированного доступа любыми методами, включая двухфакторную аутентификацию
Все данные передаваемые между клиентом и сервером должны быть зашифрованы протоколом данных https для обеспечения безопасности
NFR-02 время отклика системы на запросы пользователя не должно превышать 3 секунды даже при максимальной нагрузке

NFR-03 система должна быть совместима с разными браузерами

NFR-04 приложение должно быть легко масштабируемым для возможности добавления нового функционала и увеличения количества пользователей.

NFR-05 Время открытия экрана со списком кампаний не должно превышать 2 секунд даже при 500 активных кампаниях. Все push-уведомления должны доставляться не позже чем через 10 секунд после события
4. **Требования к пользовательскому интерфейсу (UI Requirements)**  
   - Общие принципы (например, соответствие Material Design и Human Interface Guidelines).  
   - Структура экранов (можно в виде списка или простой схемы).  
   - Ключевые элементы (кнопки, формы, модальные окна).  
   - Требования к адаптивности и доступности (размер шрифта, контрастность).  
UI-01 Цветовая схема пользовательского интерфейса должна соответствовать 

UI-02 важные уведомления системы должны отображаться на всех страницах в виде всплывающего окна или уведомления

UI-03 все элементы управления должны иметь понятное описание и всплывающие подсказки для удобства пользователей

UI-04 размер и шрифт должны быть настроены таким образом чтобы обеспечивать хорошую читабельность на разных устройствах

UI-05 интерфейс приложения должен быть адаптивным и корректно отображаться на устройствах с разным разрешением экрана

UI-06 На главном экране блогера внизу — табы: “Предложения”, “Мои сделки”, “Соцсети”, “Профиль”. При нажатии на оффер открывается карточка с бюджетом, описанием товара и кнопками “Принять” / “Отклонить”
5. **Требования к данным (Data Requirements)**  
   - Какие данные кэшируются локально.  
   - Стратегия синхронизации с сервером.  
   - Объёмы данных, пагинация.  
DR-01 вся информация о заказах должна быть доступна для отчетности через субд храниться в базе данных в течение не менее 5 лет для возможности анализа и составления отчетности.

DR-02 для всех изменений данных должен вестись журнал, фиксирующий дату, время, пользователя совершившего изменение

DR-03 данные о пользователях должны храниться в защищенной базе данных с регулярным резервным копированием

DR-04 Локально кэшируются список активных сделок и названия кампаний (обновление при каждом открытии). Фото товаров не кэшируются — подгружаются из сети. При отсутствии интернета показывается последняя актуальная версия с пометкой “Офлайн”

DR-05 структура бд должна быть оптимизирована для быстрого выполнения запросов и эффективного хранения информации
6. **Ограничения и допущения (Constraints & Assumptions)**  
   - Технические ограничения (например, приложение должно работать через существующий бэкенд, который мы не меняем).  
   - Бюджетные и временные ограничения.  
   - Допущения, которые вы сделали при формулировке требований (например, «предполагаем, что все пользователи имеют стабильный интернет большую часть времени»).
